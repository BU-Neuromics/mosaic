### 3.1 Design Philosophy

Hippo's data model is **config-driven and graph-shaped**. Entity types, their fields, and the
relationships between them are defined in a YAML or JSON schema file — not hardcoded. The
underlying storage is relational (SQLite in v0.1, PostgreSQL in future tiers), but the SDK
exposes a graph-traversal API that hides join complexity from callers. A graph database backend
(e.g., Amazon Neptune) is a valid future adapter — the query API is identical regardless of
backend.

Every entity type in Hippo shares a common structure:

- **System fields** — a minimal set of fields managed automatically by Hippo, not declarable
  in schema config
- **Status** — a single lifecycle field retained on the entity table for query performance
- **User-defined fields** — typed fields declared in schema config
- **Properties** — a JSON escape hatch for arbitrary additional metadata

Temporal metadata (`created_at`, `updated_at`, `schema_version`) is owned exclusively by the
**provenance system** (see sec6_provenance.md) and exposed as computed properties on entity
objects by the SDK. These values are never stored as columns on entity tables. The provenance
log is the single authoritative source of all temporal and audit information.

---

### 3.2 System Fields

Hippo automatically manages the following fields on every entity type. They are not declared
in schema config and cannot be overridden by users.

| Field | Type | Storage location | Writable by user |
|---|---|---|---|
| `id` | UUID | Entity table column | No — generated by Hippo on creation |
| `status` | enum | Entity table column | No — managed via lifecycle operations only |
| `created_at` | datetime (UTC) | Provenance log — computed | No |
| `updated_at` | datetime (UTC) | Provenance log — computed | No |
| `schema_version` | string | Provenance log — computed | No |
| `properties` | JSON | Entity table column | Yes — arbitrary metadata escape hatch |

**On computed fields:** `created_at` is the timestamp of the first provenance event for an
entity. `updated_at` is the timestamp of the most recent provenance event. `schema_version`
is the schema config version recorded on the most recent provenance event. All three are
derived at read time by the SDK from the provenance log and presented as first-class fields
on entity response objects — callers never need to query the provenance log directly for these
values. See sec6_provenance.md for the full provenance event model.

**There are no hard deletes in Hippo.** Status transitions replace deletion. All entities and
all provenance history are retained permanently.

### 3.3 Entity Lifecycle Status

Every entity carries a `status` field governing whether it appears in default query results.
`status` is stored as a real column on every entity table — the only state field that is —
because it is a filter condition on virtually every query and must be indexed for performance.

Status transitions are always recorded as `StatusChanged` provenance events carrying the
previous status, new status, actor, timestamp, and an optional reason string. The `status`
column is a denormalized projection of the most recent `StatusChanged` event, explicitly
maintained as such.

| Status | Meaning | Included in default queries |
|---|---|---|
| `active` | Exists, valid, fully queryable | Yes |
| `archived` | No longer active; preserved for historical reference | No |
| `superseded` | Replaced by another entity; carries `superseded_by` system relationship | No |
| `deleted` | No longer exists from the perspective of normal operations | No |
| `distributed` | Physical material sent externally; no longer in local custody | No |
| `removed` | Actively withdrawn from access or distribution | No |

**Default query behavior:** all queries filter `status = 'active'` unless the caller
explicitly requests other statuses. A **partial index** on `status = 'active'` is created
automatically by `hippo migrate` for every entity table:

```sql
CREATE INDEX idx_{entity_type}_active
ON {entity_type}s (id)
WHERE status = 'active';
```

This index contains only active records. As entities transition to non-active statuses over
time the index stays small and query performance remains constant regardless of total table
size. Historical queries that need non-active records perform a full scan, which is appropriate
since they are explicitly exceptional operations.

**WorkflowRun status note:** WorkflowRun execution states (`pending`, `running`, `succeeded`,
`failed`) are represented in the `properties` JSON field rather than as lifecycle status
values. The entity lifecycle `status` on a WorkflowRun reflects whether the run record itself
is active, archived, or superseded — not whether the pipeline completed successfully. This
keeps the lifecycle enum consistent across all entity types.

**`superseded_by` system relationship:** when an entity is superseded, a built-in
`superseded_by` edge is written to the relationships table. Setting `superseded_by`
automatically transitions the source entity's status to `superseded`. The inverse
(`supersedes`) is a computed relationship — queryable but not stored as a separate edge.
Supersession is an atomic SDK operation:

```python
client.supersede(
    entity_type="Sample",
    old_id="abc-123",
    new_id="def-456",
    actor="pipeline-run-789",
    reason="Corrected tissue region annotation"
)
```

This writes atomically: a `StatusChanged` provenance event on `abc-123` (active →
superseded) and a `superseded_by` relationship edge — or rolls back entirely on failure.

### 3.4 External IDs

Every entity can carry zero or more external identifiers from upstream systems, stored in a
shared `external_ids` table:

```
external_ids:
  id            UUID        primary key
  entity_id     UUID        indexed
  entity_type   string      indexed
  system        string      indexed    (e.g. "starlims", "halo", "nextflow")
  external_id   string      indexed
  created_at    datetime               (written by provenance system)
```

A Sample might carry a STARLIMS ID, a lab-internal barcode, and a Nextflow run ID — all
pointing to the same Hippo UUID. Lookups by external ID are a first-class query operation.
External ID records are immutable once written; correction creates a new record rather than
modifying an existing one.

### 3.5 Field Type System

| Type | Description | Example |
|---|---|---|
| `string` | UTF-8 text, optional max length | `"APOE4"` |
| `int` | Integer | `72` |
| `float` | Floating point | `1.85` |
| `bool` | Boolean | `true` |
| `date` | Calendar date (ISO 8601) | `"2021-04-15"` |
| `datetime` | UTC timestamp | `"2021-04-15T10:30:00Z"` |
| `enum` | String constrained to a declared value set | `"RNASeq"` |
| `json` | Arbitrary JSON object or array | `{"key": "value"}` |
| `uri` | S3 URI, local path, or HTTPS URL | `"s3://bucket/key"` |
| `ref` | Foreign reference to another Hippo entity by UUID | `"sample:abc-123"` |

### 3.6 Schema Config Format

#### Authoring formats

Hippo schema config is authored in the **Hippo DSL** — a concise YAML or JSON format designed
for readability and hand-authoring. At migration time the DSL is compiled transiently to
**LinkML** as an intermediate representation, which drives code generation (Pydantic models,
SQLAlchemy ORM models, JSON Schema for API validation) and provides interoperability with
other scientific data systems.

Users who prefer to author in LinkML directly may do so — Hippo accepts both formats,
auto-detected by a `format` field in the file header or by filename convention
(`*_linkml.yaml`).

```yaml
# hippo.yaml
schema:
  path: ./schema.yaml           # Hippo DSL — compiled transiently at migrate time
  # or:
  path: ./schema_linkml.yaml    # LinkML directly — no compilation step
```

Both YAML and JSON are valid for Hippo DSL. The config loader accepts `.yaml`, `.yml`, and
`.json` extensions.

**`hippo compile-schema`** explicitly outputs compiled LinkML for users who want it as a
versioned artifact for documentation or downstream tooling:

```bash
hippo compile-schema schema.yaml --out schema_linkml.yaml
```

#### Entity type declaration (Hippo DSL)

```yaml
version: "1.0"
format: hippo-dsl        # optional — auto-detected if absent

entities:
  Subject:
    description: "A biological subject who contributed one or more samples."
    base: BaseEntity
    fields:
      external_id:
        type: string
        required: true
        indexed: true
      species:
        type: enum
        values: [Homo sapiens, Mus musculus, Rattus norvegicus]
        required: true
      biological_sex:
        type: enum
        values: [male, female, unknown]
      age_at_collection:
        type: float
        description: "Age in years at time of sample collection"
      diagnosis:
        type: string
        indexed: true
```

#### Relationship declaration

```yaml
relationships:
  - name: donated
    from: Subject
    to: Sample
    cardinality: one-to-many
    description: "A subject donated one or more samples"

  - name: derived_from
    from: Sample
    to: Sample
    cardinality: many-to-many
    description: "A sample derived from another (e.g. dissection, aliquot)"
    properties:
      method: {type: string}

  - name: generated
    from: Sample
    to: Datafile
    cardinality: one-to-many

  - name: input_to
    from: Datafile
    to: WorkflowRun
    cardinality: many-to-many

  - name: output_of
    from: Datafile
    to: WorkflowRun
    cardinality: many-to-many

  - name: instance_of
    from: WorkflowRun
    to: Workflow
    cardinality: many-to-one
    required: true

  - name: contains
    from: Dataset
    to: Datafile
    cardinality: many-to-many
```

#### Schema inheritance

```yaml
entities:
  BrainSample:
    base: Sample
    description: "A sample from a brain tissue collection"
    fields:
      brain_region: {type: string, indexed: true}
      hemisphere:
        type: enum
        values: [left, right, bilateral, unknown]
      post_mortem_interval_hours: {type: float}
```

### 3.7 Core Entity Types

The following six entity types are defined in Hippo's bundled default schema
(`hippo/schema/default_schema.yaml`). Fields listed are **user-defined fields only** —
system fields (3.2) and `status` (3.3) are present on all entities implicitly and are not
repeated here.

#### Subject

A biological subject who contributed one or more samples.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `external_id` | string | yes | yes | ID from upstream subject registry |
| `species` | enum | yes | no | Biological species |
| `biological_sex` | enum | no | no | |
| `age_at_collection` | float | no | no | Years |
| `diagnosis` | string | no | yes | Primary clinical diagnosis |
| `properties` | json | no | no | Additional clinical/demographic metadata |

#### Sample

A piece of biological material derived from a Subject. May be derived from another Sample.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `external_id` | string | yes | yes | ID from upstream sample registry |
| `tissue_type` | string | yes | yes | Tissue of origin |
| `tissue_region` | string | no | yes | Sub-region within tissue |
| `collection_date` | date | no | no | |
| `passage` | int | no | no | For cell lines |
| `properties` | json | no | no | |

#### Datafile

A file at a known location containing data derived from one or more Samples.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `uri` | uri | yes | yes | S3 URI, local path, or HTTPS URL |
| `file_type` | enum | yes | yes | fastq, bam, vcf, tsv, h5ad, idat, … |
| `modality` | enum | yes | yes | RNASeq, WGBS, WGS, ATAC, genotyping, … |
| `file_size_bytes` | int | no | no | |
| `checksum_md5` | string | no | no | Integrity verification |
| `read_count` | int | no | no | For sequencing files |
| `genome_build` | string | no | yes | e.g. GRCh38 |
| `is_primary` | bool | no | yes | True if raw/unprocessed |
| `properties` | json | no | no | |

#### Dataset

A named, versioned logical collection of Datafiles.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `name` | string | yes | yes | Human-readable name |
| `version` | string | yes | no | Semver or date string |
| `description` | string | no | no | |
| `is_public` | bool | no | yes | Placeholder for future auth layer |
| `properties` | json | no | no | |

#### Workflow

A versioned analysis pipeline definition from which WorkflowRuns are instantiated.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `name` | string | yes | yes | Pipeline name |
| `version` | string | yes | yes | Semver or git SHA |
| `description` | string | no | no | |
| `repository_uri` | uri | no | no | Git repository of pipeline definition |
| `language` | enum | no | no | nextflow, snakemake, wdl, cwl, … |
| `properties` | json | no | no | |

#### WorkflowRun

A single execution of a Workflow, consuming input Datafiles and producing output Datafiles.
The `instance_of` relationship links a WorkflowRun to its Workflow definition.
Execution state (`pending`, `running`, `succeeded`, `failed`) is stored in `properties`.

| Field | Type | Required | Indexed | Notes |
|---|---|---|---|---|
| `executor` | string | no | no | Nextflow run ID, AWS Batch job ID, etc. |
| `started_at` | datetime | no | no | |
| `completed_at` | datetime | no | no | |
| `parameters` | json | yes | no | Full parameter set used for this run |
| `properties` | json | no | no | Includes execution state |

```
                    ┌─────────────┐

                    └──────┬──────┘
                           │ donated (1:many)
                    ┌──────▼──────┐
               ┌────│   Sample    │◄──────────────┐
               │    └──────┬──────┘  derived_from  │
               │           │         (many:many,   │
               │           │          self-ref)    │
               │           │ generated             │
               │    ┌──────▼──────┐           ┌───┴─────────┐
               │    │  Datafile   │──────────►│  Datafile   │
               │    └──────┬──────┘ output_of  └─────────────┘
               │           │ input_to
               │    ┌──────▼──────────────────────────────┐

               │    ┌──────▼──────┐
               │    │  Workflow   │
               │    │  (name,     │

               │    ┌─────────────┐

                    │  (contains  │
                    │  Datafiles) │
                    └─────────────┘

  [any entity] ──superseded_by──► [any entity]
```

### 3.9 Relationships Storage Model

Relationships are stored in a single `entity_relationships` edge table. New relationship
types added via schema config require no table migrations.

```
entity_relationships:
  id              UUID      primary key
  from_id         UUID      indexed
  from_type       string    indexed
  to_id           UUID      indexed
  to_type         string    indexed
  relationship    string    indexed    (e.g. "donated", "derived_from", "superseded_by")
  properties      JSON                 (relationship-level attributes)
  status          enum                 (active | removed)
  created_at      datetime             (written by provenance system)
```

Relationships are immutable once written. To remove a relationship its `status` is set to
`removed` and a `RelationshipRemoved` provenance event is written. Cardinality constraints
are enforced by the SDK at write time, not by database constraints.

### 3.10 Schema Versioning and Migration

Every schema config carries a `version` string. Hippo tracks the deployed version in
`hippo_meta`. When versions differ, `hippo migrate` must run before the server starts.

`hippo migrate` steps:
1. If input is Hippo DSL, compile transiently to LinkML
2. Diff compiled schema against deployed version in `hippo_meta`
3. Generate and print migration plan for human review
4. On confirmation (or `--yes` flag), apply and update `hippo_meta`
5. Write a `MigrationApplied` provenance event

**Migration rules:**

| Change type | Action |
|---|---|
| New entity type | Create table + partial index on `status = 'active'` |
| New field | Add nullable column (or with default if provided) |
| New relationship type | No table change — edge table is generic |
| New index | Create index |
| New enum value | Extend enum constraint |
| Field type change | **Reject** — requires manual migration |
| Remove field | Mark deprecated; column retained; excluded from API responses |
| Remove entity type | **Reject** — all entities must reach non-active status first |

Hippo never drops columns or tables automatically.

### 3.11 Validation Rules

```yaml
fields:
  modality:
    type: enum
    values: [RNASeq, WGBS, ATAC, WGS, genotyping]
    required: true
  uri:
    type: uri
    required: true
    validators:
      - type: uri_scheme
        allowed: [s3, file, https]
  read_count:
    type: int
    validators:
      - type: range
        min: 0
  external_id:
    type: string
    required: true
    validators:
      - type: unique
        scope: entity_type
```

Built-in validators for v0.1: `required`, `unique`, `range`, `uri_scheme`, `regex`,
`min_length`, `max_length`. Custom validators register via the plugin system as Python
callables.

### 3.12 Adding Custom Entity Types

```yaml
entities:
  CellLine:
    description: "An immortalized cell line derived from a subject"
    base: Sample
    fields:
      cell_line_name: {type: string, required: true, indexed: true}
      passage_number: {type: int}
      culture_conditions: {type: json}

relationships:
  - name: established_from
    from: CellLine
    to: Sample
    cardinality: many-to-one
```

Run `hippo migrate` to apply. The new entity type is immediately queryable and ingestable
with no further changes.

### 3.13 Multi-tenancy and Namespacing

Single namespace in v0.1. `is_public` on Dataset is a placeholder for future auth-layer
enforcement. Multi-tenancy is explicitly out of scope for v0.1.
