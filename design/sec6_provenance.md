## 6. Provenance & Audit

**Document status:** Draft v0.1
**Depends on:** sec3_data_model.md, sec3b_relational_storage.md
**Feeds into:** sec4_api_layer.md, sec5_ingestion.md

### 6.1 Design Philosophy

Provenance is a first-class, non-optional feature of Hippo. Every write to any entity,
relationship, or external ID is recorded as an immutable provenance event. There are no hard
deletes — only availability transitions. The provenance log is the single authoritative source
of truth for:

- When an entity was created and last modified
- Who made each change and why
- What the entity's state was at any point in time
- What system or workflow produced or modified the data

Provenance records are **append-only** and **never modified**. Once written, they cannot be
updated, redacted, or deleted by any normal SDK operation.

---

### 6.2 Provenance Event Model

Every write operation produces one or more `ProvenanceEvent` records. Events are immutable
value objects stored in a dedicated provenance table (see sec3b §3b.6).

```python
@dataclass(frozen=True)
class ProvenanceEvent:
    id: str                     # UUID, generated by Hippo
    event_type: str             # one of the event types in §6.3
    entity_id: str              # UUID of the affected entity
    entity_type: str            # entity type name (e.g. "Sample")
    timestamp: datetime         # UTC, set by Hippo at write time — not caller-supplied
    actor: str                  # string identity of the writer (user, pipeline, system)
    schema_version: str         # deployed schema config version at time of write
    diff: dict | None           # field-level change diff for EntityUpdated events
    context: dict | None        # structured provenance context (see §6.5)
    detail: dict | None         # event-type-specific metadata
```

**Immutability:** The `timestamp` field is always set by Hippo at write time. Callers cannot
supply or override timestamps. This ensures the audit log cannot be backdated.

**`diff` field:** For `EntityUpdated` events, `diff` contains a field-level change record:

```json
{
  "tissue_type": {"from": "cortex", "to": "hippocampus"},
  "collection_date": {"from": null, "to": "2024-03-01"}
}
```

Fields not present in `diff` were unchanged. System fields (`id`, `is_available`) have their
own dedicated event types and are not recorded in `diff`.

---

### 6.3 Event Types

| Event type | Triggered by | `detail` payload |
|---|---|---|
| `EntityCreated` | First `put()` for a new entity ID | `{"initial_state": {...}}` |
| `EntityUpdated` | Subsequent `put()` for an existing entity | `diff` field populated |
| `AvailabilityChanged` | `set_availability()` SDK call | `{"from": true, "to": false, "reason": "..."}` |
| `RelationshipCreated` | Creating a new relationship edge | `{"relationship": "...", "from_id": "...", "to_id": "..."}` |
| `RelationshipRemoved` | Setting relationship status to `removed` | same as above |
| `ExternalIdAdded` | Adding an external ID mapping | `{"system": "...", "external_id": "..."}` |
| `ExternalIdSuperseded` | Correcting an external ID | `{"old_record_id": "...", "new_record_id": "..."}` |
| `SupersededBy` | `supersede()` SDK call | `{"new_entity_id": "...", "reason": "..."}` |
| `MigrationApplied` | `hippo migrate` completion | `{"from_version": "...", "to_version": "..."}` |
| `ReferenceInstalled` | `hippo reference install` | `{"loader": "...", "version": "..."}` |

**Atomic events:** Compound SDK operations write multiple events atomically. For example,
`client.supersede(old_id, new_id)` writes an `AvailabilityChanged` event on `old_id`
(is_available: true → false) and a `SupersededBy` event on `old_id` in a single transaction.

### 6.4 Provenance Table Schema

```sql
CREATE TABLE provenance_events (
    id           TEXT    PRIMARY KEY,    -- UUID
    event_type   TEXT    NOT NULL,
    entity_id    TEXT    NOT NULL,
    entity_type  TEXT    NOT NULL,
    timestamp    TEXT    NOT NULL,       -- ISO 8601 UTC
    actor        TEXT    NOT NULL,
    schema_ver   TEXT    NOT NULL,
    diff         TEXT,                   -- JSON string or NULL
    context      TEXT,                   -- JSON string or NULL
    detail       TEXT                    -- JSON string or NULL
);

CREATE INDEX idx_provenance_entity ON provenance_events (entity_id, entity_type);
CREATE INDEX idx_provenance_timestamp ON provenance_events (timestamp);
CREATE INDEX idx_provenance_actor ON provenance_events (actor);
```

The `idx_provenance_timestamp` index supports Cappella's polling trigger
(`updated_at > watermark` queries) without requiring a denormalised column on entity tables.
See §6.7 for the polling query pattern.

### 6.5 Structured Provenance Context

The `actor` field accepts a plain string identity for simple cases (e.g. `"admin"`,
`"pipeline-run-42"`). For richer attribution — particularly from Cappella workflow runs —
callers may supply a `provenance_context` dict via `WriteOperation.provenance_context`.
This is stored verbatim in the `context` column.

**Recommended context schema for Cappella workflow runs:**

```json
{
  "source": "cappella",
  "trigger": "starlims_new_specimen",
  "adapter": "starlims",
  "sync_run_id": "uuid-of-sync-run",
  "workflow_name": "qc_pipeline",
  "workflow_run_id": "nextflow-run-abc123"
}
```

Hippo does not validate or enforce the shape of `context` — it is stored as-is and made
queryable via the provenance API. The structured context enables questions like:
"show me all entities written by the `qc_pipeline` workflow in the last 30 days."

**Opinionated decision:** Hippo does not define a canonical provenance context schema. Context
shape is the caller's (Cappella's) responsibility. Hippo's provenance API allows filtering
provenance events by any top-level key in the `context` JSON.

### 6.6 Computed Temporal Fields

The system fields `created_at`, `updated_at`, and `schema_version` on entity response objects
are not stored in entity tables. They are derived from the provenance log at read time:

| Field | Derivation |
|---|---|
| `created_at` | `MIN(timestamp) WHERE entity_id = ? AND event_type = 'EntityCreated'` |
| `updated_at` | `MAX(timestamp) WHERE entity_id = ?` (any event type) |
| `schema_version` | `schema_ver` from the most recent provenance event for the entity |

**Batch optimisation:** The storage adapter may materialise a provenance summary view for
efficient batch reads:

```sql
CREATE VIEW entity_temporal AS
SELECT
    entity_id,
    entity_type,
    MIN(timestamp) FILTER (WHERE event_type = 'EntityCreated') AS created_at,
    MAX(timestamp) AS updated_at,
    (SELECT schema_ver FROM provenance_events p2
     WHERE p2.entity_id = p.entity_id ORDER BY timestamp DESC LIMIT 1) AS schema_version
FROM provenance_events p
GROUP BY entity_id, entity_type;
```

Whether to use a view, a subquery, or a denormalised summary table is an adapter
implementation choice. All approaches produce identical results at the SDK level.

### 6.7 Provenance API

The `HippoClient` exposes provenance as a first-class query surface:

```python
# Full history for a single entity
client.history("Sample", entity_id)
# → list[ProvenanceEvent], ordered oldest-first

# Events for a type since a timestamp (used by Cappella hippo_poll trigger)
client.events_since("Sample", since=datetime(...), event_types=["EntityCreated"])
# → list[ProvenanceEvent]

# Events by actor or context key
client.events_by_actor("cappella", since=datetime(...))
client.events_by_context("sync_run_id", "uuid-of-sync-run")
# → list[ProvenanceEvent]

# Point-in-time snapshot of an entity
client.entity_at("Sample", entity_id, at=datetime(...))
# → dict (reconstructed state by replaying events up to `at`)
```

**Point-in-time reconstruction** (`entity_at`) replays `EntityCreated` + `EntityUpdated`
diffs in order to reconstruct the entity state at any past timestamp. This is a read-only
operation — it does not write any provenance record.

**REST endpoints:**

| Method | Path | Description |
|---|---|---|
| `GET` | `/provenance/{entity_type}/{entity_id}` | Full event history for an entity |
| `GET` | `/provenance/{entity_type}/{entity_id}/at?t=<iso>` | Point-in-time snapshot |
| `GET` | `/provenance/events?since=<iso>&actor=<s>&event_type=<t>` | Query events by filter |

### 6.8 Retention Policy

**Opinionated decision:** Provenance records are retained indefinitely. There is no automated
retention policy or TTL in v0.1.

Rationale: the provenance log is the sole source of `created_at` / `updated_at` for entities.
Truncating it would destroy temporal metadata. In research contexts, long-term audit trails
are a regulatory and reproducibility requirement.

**Open question:** A future "provenance archival" feature could move older events to cold
storage (S3 Glacier, etc.) while retaining a summary record. Not in scope for v0.1.

### 6.9 ProvenanceManager

The `ProvenanceManager` in `hippo/core/provenance.py` is an internal SDK component. It is
not part of the public API. Its responsibilities:

- Write provenance events as part of every `EntityStore.put()` transaction
- Derive `created_at`, `updated_at`, `schema_version` for entity read responses
- Expose the provenance query methods on `HippoClient`
- Validate that events are written atomically with the entity change they record

The `ProvenanceManager` is injected into `HippoClient` at construction time. Storage adapters
implement the provenance write and query operations alongside entity operations — there is no
separate provenance adapter.
